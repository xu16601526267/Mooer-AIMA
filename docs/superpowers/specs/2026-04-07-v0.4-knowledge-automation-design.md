# AIMA v0.4 设计规格 — 知识自动化与中心化知识管理

> **版本**: v0.4 "Knowledge Autonomy"
> **日期**: 2026-04-07
> **状态**: 设计完成，待实现

## 1. 目标

v0.3.0 完成了知识获取的工具层（56 MCP 工具、Benchmark Runner、Tuner、ExplorationManager、Patrol、Central Server）。但这些组件需要人工触发，缺乏自主运行能力。

v0.4 的核心目标：**让 AIMA 自主产生新知识**。

- 边缘设备连接云端 LLM 时（Tier 2），具备完整的自主探索能力：发现缺口 → 生成假设 → 验证 → 收割成果 → 同步
- 边缘设备使用本地小模型时（Tier 1），具备基础的确定性探索能力
- 中央云服务器具备超强模型推理 + 分析能力，结合 Scenario 给出优质配置参数

## 2. 架构方案：分层智能

三级能力递进，每级做它最擅长的事：

```
Tier 3: 中央（全局视野）                    ←── 云端大模型 (Claude/GPT-4)
   · 跨设备 benchmark 模式分析
   · 生成/优化 Scenario
   · 为新 HW×Model 推导配置
   · 定期后台全局分析

Tier 2: 边缘 L3a（设备视野 + 云端推理）     ←── 云端大模型 API
   · 自治循环：gaps → 假设 → benchmark → promote
   · 事件驱动：部署后/OOM/新模型 → 触发探索
   · 向中央请求推荐配置
   · 复杂决策（参数权衡、引擎选择推理）

Tier 1: 边缘本地模型（设备视野 + 本地推理） ←── 本地小模型 (Qwen3-8B 等)
   · 确定性探索：gaps → benchmark 最常见缺口
   · 简单事件响应：OOM → 降 gmu → 重部署
   · 本地知识整理：格式化 benchmark 结果、生成摘要
   · 不做复杂推理，不生成 scenario
```

### 2.2 层级定义

| 层级 | 触发条件 | 能力范围 | LLM 要求 | 离线可用 |
|------|---------|---------|----------|---------|
| **Tier 1** | 本地有任意 LLM | 确定性探索（gaps→benchmark→promote）、简单事件响应、知识整理 | 本地小模型 | 完全离线 |
| **Tier 2** | 连接到云端强 LLM | Tier 1 全部 + 假设生成、自适应调优、引擎选择推理、向中央请求推荐 | 云端大模型 | 降级到 Tier 1 |
| **Tier 3** | Central Server 运行中 | 跨设备模式分析、scenario 生成/优化、全局配置推荐、定期分析报告 | 云端大模型 | 服务端 |

### 2.3 降级策略

```
Agent 启动时自动探测能力等级：

  ┌─ LLM 端点可达？
  │    ├─ Yes → 检测模型能力
  │    │         ├─ 支持 tool_calling + 大上下文 → Tier 2
  │    │         └─ 仅 context_only 或小模型   → Tier 1
  │    └─ No  → Tier 0 (仅 Patrol，无主动探索)
  │
  └─ Central 可达？
       ├─ Yes → 启用 sync + advisory 请求
       └─ No  → 纯本地运行（成果暂存，等恢复后 sync）
```

降级是无缝的。Tier 2 运行中如果云端 LLM 断连，自动降级到 Tier 1 继续执行当前探索计划中的确定性任务，跳过需要强推理的步骤。每 5 分钟重试 LLM 连接，恢复后自动升级。

### 3.2 Scheduler：两种触发机制

**定时循环（后台自治）：**

```go
type ScheduleConfig struct {
    GapScanInterval    time.Duration  // 默认 24h — 扫描知识缺口
    FullAuditInterval  time.Duration  // 默认 7d — 全量知识审计
    SyncInterval       time.Duration  // 默认 6h — 与中央同步
    MaxConcurrentRuns  int            // 默认 1 — 同时进行的探索数
    QuietHours         [2]int         // 默认 [2,6] — 凌晨2-6点不探索
}
```

**事件驱动（即时响应）：**

| 事件 | 来源 | 触发的探索 |
|------|------|-----------|
| `deploy.completed` | 部署成功后 | 自动跑一轮 benchmark，建立基线 |
| `patrol.alert.oom` | Patrol 发现 OOM | tune gmu 参数（Tier 1 可做） |
| `patrol.alert.idle` | GPU 长时间空闲 | 趁空闲跑 gaps 填补 |
| `model.discovered` | 发现新模型文件 | resolve + benchmark 新模型 |
| `central.advisory` | 中央推送推荐配置 | 本地验证推荐配置 |
| `central.scenario` | 中央推送新 scenario | 本地验证 scenario |

### 3.3 Planner：Tier 1 vs Tier 2

```go
type Planner interface {
    Plan(ctx context.Context, input PlanInput) (*ExplorationPlan, error)
}

type PlanInput struct {
    Hardware      HardwareInfo
    Gaps          []GapEntry
    ActiveDeploys []DeployStatus
    Advisories    []Advisory
    History       []ExplorationRun
    OpenQuestions []OpenQuestion
    Event         *ExplorerEvent       // nil = 定时触发
}

type ExplorationPlan struct {
    ID        string
    Tier      int
    Tasks     []PlanTask
    Reasoning string   // Tier 2: LLM 推理过程; Tier 1: "rule-based"
}

type PlanTask struct {
    Kind      string            // "validate" / "tune" / "open_question" / "compare"
    Model     string
    Engine    string
    Params    map[string]any
    Reason    string
    Priority  int
    DependsOn string
}
```

**Tier 1 RulePlanner** — 固定优先级规则：

1. 已部署但无 benchmark → 最高优先（建立基线）
2. 中央 advisory → 验证推荐配置
3. gaps 按模型热度排序 → 填补（每轮最多 3 个）
4. golden config 测试次数 < 3 → tune 强化
5. open_questions untested → 验证

**Tier 2 LLMPlanner** — LLM 分析后生成智能计划：

- "这个模型有 AWQ 和 GPTQ 两种量化，应该都测一下对比"
- "中央推荐 gmu=0.78，但本地 VRAM 比推荐的设备大，试试 0.85"
- "上次 tune 发现 tp=2 最优，这次换个模型也先从 tp=2 开始"
- "这个 open_question 需要先部署 X 再部署 Y 才能验证共存"

### 3.4 Harvester：成果收割

**Tier 1 模式：**
- maybeAutoPromote() (>5% 升级 golden)
- 模板化 knowledge_note: `"{model} on {engine}: {throughput} tok/s, {gmu} optimal"`
- open_question 更新 status + actual_result

**Tier 2 额外能力：**
- LLM 生成深度洞察（"AWQ 在 W7900D 上比 BF16 快 23%，但 TTFT 增加 15%"）
- LLM 调整后续计划（"tp=4 反而慢，取消 tp=8 测试"）
- LLM 判断是否需要向中央反馈

### 3.4.1 Benchmark-Backed Knowledge Contract

v0.4 的“知识”不能只是一段 note 文本。凡是 Explorer/Harvester 认定为成功探索并持久化的结果，必须同时落下可追溯的 benchmark 证据，并满足下面的最小字段集：

- **身份上下文**
  - `hardware_profile` / `gpu_arch`
  - `model`
  - `engine`（asset ID）
  - `engine_version`，以及容器引擎的 `engine_image`（若可得）
  - `benchmark_id` / `config_id`

- **部署配置**
  - 完整 resolved config，不能退化成 benchmark profile
  - 需要保留高级参数，例如 `tp_size`、`mem_fraction_static`、`max_running_requests`
  - 对异构引擎必须保留 offload 相关参数，例如 `kt_cpuinfer`、`kt_num_gpu_experts`、`kt_threadpool_count`、`n_gpu_layers`

- **Benchmark Profile**
  - `concurrency`
  - `num_requests`
  - `warmup_count`
  - `rounds`
  - `input_tokens`
  - `max_tokens`
  - `avg_input_tokens`
  - `avg_output_tokens`

- **性能指标**
  - `ttft_p50_ms` / `ttft_p95_ms` / `ttft_p99_ms`
  - `tpot_p50_ms` / `tpot_p95_ms`
  - `throughput_tps`
  - `qps`
  - `error_rate`
  - `sample_count`
  - `duration_s`
  - `stability`

- **资源观测**
  - `vram_usage_mib`
  - `ram_usage_mib`
  - `gpu_utilization_pct`
  - `cpu_usage_pct`
  - `power_draw_watts`

- **资源观测口径**
  - `vram_usage_mib` / `ram_usage_mib` 记录 benchmark 窗口内峰值
  - `gpu_utilization_pct` / `cpu_usage_pct` / `power_draw_watts` 记录 benchmark 窗口内均值
  - response / DB / overlay / note 必须共享同一口径，不能混用结束瞬时快照

- **异构引擎补充要求**
  - 对 KTransformers / sglang-kt / llama.cpp offload 这类 CPU+GPU 混合路径，知识里必须显式记录 CPU 与 RAM 是否参与推理，而不是只记录 GPU 指标
  - 推荐在 overlay 中保留 `heterogeneous_observation`，至少包含 CPU 使用率、RAM 使用量、以及关键 offload 参数
  - `heterogeneous_observation` 优先由 catalog/compat 事实驱动，缺失时才允许用 deploy config + benchmark 资源证据保守推断

持久化形态要求：

- `benchmark_results` 是结构化事实源，字段必须完整。
- `catalog/models/*.yaml` overlay 的 `expected_performance` 需要同时保留 resolver 可消费的平铺字段，以及面向分析的 `benchmark_profile` / `resource_usage` 结构化块。
- `knowledge_notes` 是解释层，必须引用上面的事实源，不能脱离 benchmark 证据自由发挥。

**运行时降级行为：**
- Planner 调用 LLM 失败 → 切换到 RulePlanner，LLM-only 任务标记 `skipped_tier_degraded`
- Harvester 调用 LLM 失败 → 降级到模板化 note，标记 `insight_pending`
- 每 5 分钟重试 LLM，恢复后补充执行 skipped 任务

### 4.2 数据库策略

边缘永远是 SQLite，中央不限定具体数据库，通过 `CentralStore` 接口解耦。

```go
type CentralStore interface {
    // 数据写入
    IngestConfigurations(ctx context.Context, configs []IngestConfig) (*IngestResult, error)
    IngestBenchmarks(ctx context.Context, benchmarks []IngestBenchmark) (*IngestResult, error)
    UpsertDevice(ctx context.Context, device DeviceInfo) error

    // 数据查询
    QueryConfigs(ctx context.Context, params QueryParams) ([]Configuration, error)
    QueryBenchmarks(ctx context.Context, params BenchmarkQuery) ([]BenchmarkResult, error)
    QueryGaps(ctx context.Context, params GapsParams) ([]GapEntry, error)

    // Advisory 管理
    InsertAdvisory(ctx context.Context, adv Advisory) error
    ListAdvisories(ctx context.Context, params AdvisoryQuery) ([]Advisory, error)
    UpdateAdvisoryStatus(ctx context.Context, id string, status string) error

    // Scenario 管理
    UpsertScenario(ctx context.Context, s Scenario) error
    ListScenarios(ctx context.Context, hardware string) ([]Scenario, error)

    // 分析任务
    InsertAnalysisRun(ctx context.Context, run AnalysisRun) error
    UpdateAnalysisRun(ctx context.Context, id string, updates AnalysisRunUpdate) error
}
```

v0.4 提供两个实现：

| 实现 | 适用场景 | 说明 |
|------|---------|------|
| `store_sqlite.go` | 开发/小规模（<10 台设备） | 零依赖，快速启动 |
| `store_postgres.go` | 生产/大规模 | 并发读写，JSONB 索引，适合海量聚合查询 |

运行时通过 `CENTRAL_DB_DRIVER` 环境变量选择。PostgreSQL 驱动使用 `github.com/jackc/pgx/v5`（纯 Go，符合零 CGO 约束）。

数据库与 LLM 推理的适配：Advisor Engine 通过 `CentralStore` 接口拿到结构化数据后，序列化为 Markdown 表格/JSON 摘要嵌入 LLM prompt。

### 4.3 Advisor Engine：三种推荐能力

**能力 1：单配置推荐 (Recommend)**

```
请求: POST /api/v1/advise
{
  "hardware_profile": "nvidia-rtx4090-x86",
  "hardware_info": { "gpu_vram_mib": 24576, "gpu_count": 1 },
  "model": "qwen3-30b-a3b",
  "engine": "",                // 空 = 也推荐引擎
  "intent": "throughput"       // throughput / latency / balanced
}

处理流程:
  1. 查询: 相同/相似硬件上该模型的所有 benchmark 历史
  2. 查询: 该硬件上所有模型的 golden configs
  3. 构造 LLM prompt: 硬件规格 + 历史数据 + 引擎默认参数 + intent
  4. LLM 推理 → 结构化输出
  5. 存储 advisory 记录

响应:
{
  "advisory_id": "adv-a1b2c3",
  "engine": "vllm",
  "config": {
    "gpu_memory_utilization": 0.78,
    "quantization": "awq",
    "max_model_len": 16384,
    "tensor_parallel_size": 1
  },
  "reasoning": "24GB VRAM 不足以 BF16 全量加载 30B MoE，AWQ 量化后约 16GB...",
  "confidence": "medium",
  "based_on": ["config-xxx", "config-yyy"],
  "suggested_validation": {
    "kind": "tune",
    "params": [{"key": "gpu_memory_utilization", "values": [0.7, 0.78, 0.85]}]
  }
}
```

**能力 2：优化已有 Scenario (OptimizeScenario)**

取某 scenario 的所有设备 benchmark 数据，LLM 分析参数优化空间（如 "gmu=0.50 过于保守，3 台设备实测 0.65 均稳定"），生成优化版参数 + 变更理由。

**能力 3：生成新 Scenario (GenerateScenario)**

根据硬件 profile + 模态需求 + 约束条件，LLM 推理资源分配、模型选择、启动顺序，生成完整 DeploymentScenario YAML + 需验证的 open_questions。

## 5. 边缘-中央协议 v2

### 5.1 Sync 协议升级

```
边缘设备                                          中央服务器

  数据上行 (不变):
    sync push ──→ configurations + benchmarks + notes

  数据下行 (增强):
    sync pull ←── configurations + benchmarks + notes
                  + advisories (新增)
                  + scenarios  (新增)

  按需请求 (新增):
    advise    ──→ "我是什么硬件，想跑什么模型" ──→ 推荐配置
    scenario/generate ──→ "给我生成部署方案" ──→ 新 scenario
    feedback  ──→ "advisory X 验证结果" ──→ 更新状态
```

### 5.2 Advisory 生命周期

```
中央生成 advisory (status: pending)
    │
    ├─ 边缘 sync pull → 获取 pending advisories → status: delivered
    │
    ├─ 边缘 Explorer 收到 central.advisory 事件
    │   ├─ 自动验证 (benchmark 推荐配置)
    │   │   ├─ 性能达标 → promote golden + 反馈中央 → status: validated
    │   │   └─ 性能不达标 → 反馈中央 + 原因 → status: rejected
    │   └─ Tier 1: 直接应用 (信任 high confidence 推荐)
    │
    └─ 过期 (30天未被拉取) → status: expired
```

```

                                                    → 查询已有 RDNA3 数据

  [5] sync pull                                ←── [5] 返回 advisories
      → 收到推荐配置

  [6] EventBus: central.advisory 事件
      → Planner 生成验证计划

  [7] Executor 执行:
      → benchmark(vllm, gmu=0.78, awq, tp=1)
      → benchmark(vllm, gmu=0.78, awq, tp=2)
```

## 6. Tier 1 vs Tier 2 详细分工

### 6.1 流水线双模式

```
Explorer 流水线:

  触发 → Scheduler/EventBus (两个 Tier 相同)
    │
    ▼
  发现 → "该探索什么？"
    │    Tier 1: gaps() 规则排序
    │    Tier 2: gaps() + LLM 分析优先级
    ▼
  计划 → "怎么探索？"
    │    Tier 1: 固定模板 (validate → tune gmu → promote)
    │    Tier 2: LLM 生成自适应计划 (选引擎、选量化、多维对比)
    ▼
  执行 → ExplorationManager (两个 Tier 相同，复用现有)
    │
    ▼
  收割 → "成果怎么处理？"
    │    Tier 1: 模板化 note + auto-promote
    │    Tier 2: LLM 生成洞察 + 智能 promote + 调整后续计划
    ▼
  同步 → sync push (两个 Tier 相同)
```

### 6.2 降级的具体行为

```
运行时降级 (Tier 2 → Tier 1):

  Planner 调用 LLM 失败
    → 日志: "LLM unavailable, degrading to Tier 1"
    → 切换到 RulePlanner
    → 当前计划中未执行的 LLM-only 任务标记为 skipped_tier_degraded
    → 继续执行确定性任务

  Harvester 调用 LLM 失败
    → 降级到模板化 note
    → 标记 insight_pending → 下次 LLM 恢复时补充分析

  每 5 分钟重试 LLM 连接
    → 恢复后自动升级回 Tier 2
    → 补充执行之前 skipped 的任务
```

### 7.3 设计约束

| 约束 | 原因 |
|------|------|
| Explorer 不直接调用 MCP 工具 | 通过 ExplorationManager/Tuner 间接调用，保持现有工具调用链 |
| Planner 不执行任何副作用 | 只生成计划，不部署/不 benchmark。执行权在 Executor |
| EventBus 是进程内的 | `chan ExplorerEvent` 足够，不引入外部消息队列 |

| PostgreSQL 是可选依赖 | 运行时通过 `CENTRAL_DB_DRIVER` 选择 |
| 新增表通过 migration 版本管理 | 边缘: migrateV8，中央: 独立 migration 链 |

## 10. 成功标准

### 10.1 E2E 验证点

v0.4 在真实机器上的端到端验收，不再只看“有没有跑起来”，而是按下面 7 个验证点判定：

1. **云端 LLM 连通性与流式可观测性**
   - `agent status` / 实测 `ask` 可确认 endpoint 可达
   - Planner 调用必须能区分 “未连上 / 首 token 前 / reasoning 流中 / content 输出中 / 超时截断”

2. **计划生成正确**
   - 计划必须基于本机真实可执行的模型与引擎，不允许把不存在的本机 asset 当能力
   - 在共享 GPU 场景中，计划至少不能持续生成明显不可执行的任务

3. **执行链路打通**
   - 计划进入 `deploy.apply` / `benchmark.run` / `tuning.start`
   - 如果已有 ready deployment，可直接 validate 而不重复 deploy
   - 如果需要释放 GPU，`deploy.delete` 的实际结果必须被检查，不能假成功后傻等

4. **Benchmark 事实完整持久化**
   - `configurations` 与 `benchmark_results` 都要落库
   - `configurations.config` 必须是真实 deploy config，不是 benchmark profile
   - `benchmark_results` 必须包含 Benchmark Profile + 延迟/吞吐 + 资源观测

5. **知识 overlay 有实际价值**
   - `catalog/models/*.yaml` overlay 必须包含 `engine_version`、高级参数、benchmark profile、资源观测
   - 对 KT/offload 引擎，overlay 必须能体现 CPU/RAM 参与推理

6. **knowledge_note grounded**
   - note 必须以 benchmark / deploy artifact 为依据
   - note 可以做总结，但不能替代结构化事实，也不能虚构失败原因

7. **持续探索而非单次跑完**
   - 第一条 validate 成功后，Explorer 能继续进入下一个高价值任务（例如 tune / compare / open_question）
   - 不允许出现 active plan 持续堆积、只计划不执行的假自治状态

### 10.2 功能性
- 新设备启动 AIMA 后，24h 内自动产生首批 benchmark + golden config
- 中央能为未见过的 HW×Model 组合生成可用配置（>70% 首次可用率）
- Tier 2→1 降级过程中不丢失已完成的探索成果
- Central PostgreSQL 和 SQLite 行为一致（同一测试套件通过）

### 10.3 运维性
- Explorer 安静时段内不占用 GPU
- Explorer 不干扰正在运行的推理服务 (MaxConcurrentRuns=1)
- Central LLM 成本可控（每个 advisory 调用 < 10K tokens）

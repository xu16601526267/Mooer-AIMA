# AIMA v0.4 Agent Team Charter

> Purpose: coordinate parallel agent implementation for `v0.4`
> Parent plan: `docs/superpowers/plans/2026-04-07-v0.4-execution-program.md`

## 1. Team Topology

### Team Lead

Responsibilities:

- keep the execution program authoritative
- assign write ownership
- review contract compatibility before merge
- run integration and release gates

### Team A: Edge Explorer

Write ownership:

- `internal/agent/explorer.go`
- `internal/agent/explorer_scheduler.go`

- `cmd/aima/main.go`

- close local autonomy loop
- enforce execution semantics
- wire live config and callbacks

### Team B: Sync and Edge Protocol

Write ownership:

- `cmd/aima/tooldeps_integration.go`
- `internal/mcp/tools_knowledge.go`
- `internal/mcp/tools_deps.go`
- `internal/cli/knowledge.go`
- `internal/cli/scenario.go`

- align edge protocol behavior with the final advisory/scenario contract

### Team C: Central Platform

### Team D: Integration and Release

Write ownership:

- integration tests
- docs updates after stream merges
- non-overlapping fixes needed for release hardening

Mission:

- turn partial stream outputs into a release candidate

## 2. Team Rules

1. Each team owns its listed files and should not edit another team's files unless reassigned.
2. No team may revert unrelated user or agent changes.
3. Contract changes proposed by Team A or B must be ratified by Team C before merge.
4. `cmd/aima/main.go` is single-owner to avoid repeated merge conflict churn.
5. Integration tests are the gate, not verbal agreement.

## 3. Merge Order

1. Team C lands contract lock changes first.
2. Team A lands local Explorer execution changes against the locked contract.
3. Team B lands sync v2 and edge protocol alignment.
4. Team D lands end-to-end tests and hardening fixes.

## 4. Handoff Requirements

Every team handoff must include:

- changed files
- contract assumptions
- new tests added
- known gaps left open
- anything another team must rebase against

## 5. Definition of a Good Parallel Task

A task is ready for parallel execution only if:

- its write set is clear
- its acceptance criteria are explicit
- it does not depend on another unfinished schema decision
- it can be reviewed without reading unrelated modules

## 6. Initial Parallelization Plan

### Round 1

- Team C: lock advisory/scenario/analysis schemas and API lifecycle.
- Team A: wire Explorer inputs, execution context, and harvester callbacks without changing central contract.
- Team B: prepare sync/CLI/MCP behavior against the agreed contract draft, but avoid final endpoint semantics until Team C lands.

### Round 2

- Team B: finalize sync v2 after Team C merges.
- Team A: finalize advisory validation and scenario intake after Team B merges.
- Team D: add full-path integration tests and UAT checklist.

### Round 3

- Team Lead: run integration, close remaining incompatibilities, prepare release notes and final docs.

## 7. Stop Conditions

The team must stop feature expansion and return to contract review if any of the following occurs:

- advisory field names diverge between edge and central
- scenario payload shape changes after sync implementation begins
- Explorer produces successful runs without durable outputs
- analyzer writes duplicate run records instead of lifecycle updates

## 8. Success Signal

The team is successful only when a clean repo clone can:

1. start central
2. start edge
3. sync baseline data
4. receive an advisory
5. validate it automatically
6. push feedback
7. show the updated lifecycle on central

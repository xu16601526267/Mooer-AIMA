## 1. Why This Document Exists

The repo already contains a complete `v0.4` design spec and three implementation plans, but the current codebase is still between `v0.3` and `v0.4`:

- core `v0.4` files and tests exist
- many interfaces are present but the data contract is not closed
- edge and central paths compile, but the autonomy loop is not yet end-to-end

This document is the execution baseline for taking the repo from "skeleton present" to "releaseable `v0.4.0` implementation".

## 2. Release Objective

`v0.4.0` is complete only when all of the following are true:

1. Edge Explorer can autonomously discover work, execute benchmark or tune tasks, harvest results, persist artifacts, and sync them upstream.
2. Central can generate advisories and scenarios with a stable schema and lifecycle.
3. Sync v2 is stateful and deterministic, not a best-effort side channel.
4. A new device can complete the full loop:
   detect hardware -> sync push -> receive advisory -> validate locally -> promote if better -> push feedback -> central updates lifecycle.

## 3. Current State Summary

### 3.1 Present in repo

- Explorer subsystem exists under `internal/agent/`.
- Central server, store abstraction, SQLite and PostgreSQL implementations exist under `internal/central/`.
- CLI and MCP entrypoints for explorer, advise, and scenario generation exist.
- Sync-side HTTP closures exist in `cmd/aima/tooldeps_integration.go`.
- The relevant packages currently pass their unit and integration tests:
  - `go test ./internal/agent/...`
  - `go test ./internal/central/...`

### 3.2 Not yet closed

- Explorer is not fully wired with real gap/deploy/open-question inputs.
- Explorer-triggered benchmark runs do not reliably persist knowledge because task execution does not carry complete hardware context.
- Harvester is not wired to save notes or sync after success.
- Advisory schema and lifecycle do not match the design spec.
- Sync v2 is implemented as extra pull requests plus in-process events, not as a first-class protocol contract.
- Central analyzer does not yet implement post-ingest scheduling, scenario health analysis, or run-state updates.
- Runtime configuration for Explorer and Central analyzer is mostly stubbed.

## 4. P0 Architecture Decisions

These decisions must be treated as locked before feature work continues.

### 4.1 Advisory contract is the primary `v0.4` boundary

All edge-central autonomy depends on a stable advisory schema. The implementation must align with the design spec:

- status: `pending`, `delivered`, `validated`, `rejected`, `expired`
- payload: structured recommendation content, not free-form summary text only
- timestamps: `created_at`, `delivered_at`, `validated_at`
- targeting: `target_hardware`, `target_model`, `target_engine`

The current `accepted bool + feedback string` shape is insufficient for `v0.4`.

### 4.2 Sync v2 is a stateful protocol, not an event bridge

`sync pull` may still internally publish events, but that is a local delivery mechanism only. The protocol contract must carry:

- what advisories are eligible for a device
- when they transition to delivered
- how feedback updates lifecycle state
- how scenarios are versioned and re-fetched safely

### 4.3 Explorer must produce durable knowledge artifacts

A completed exploration is not useful unless it can leave behind at least one of:

- benchmark result
- configuration status update
- knowledge note
- open question resolution
- sync push payload

### 4.4 Central analyzer is a lifecycle manager, not just an LLM caller

Analyzer must own:

- scheduled gap scans
- scheduled pattern discovery
- scheduled scenario health checks
- post-ingest delayed analysis
- proper `analysis_runs` state transitions

## 5. Workstreams

### Stream A: Edge Explorer Closure

Owner scope:

- `internal/agent/explorer*.go`
- `internal/agent/patrol.go`
- `cmd/aima/main.go`

Primary deliverables:

1. Wire real inputs into `PlanInput`:
   - hardware
   - gaps
   - active deployments
   - advisories
   - open questions
   - recent exploration history
2. Carry hardware and recommended params into execution so benchmark runs persist knowledge.
3. Make Tier 1 and Tier 2 degrade and recover in the exact ways promised by the spec.
4. Make `explorer.config` mutate live schedule state instead of returning a stubbed success.
5. Ensure `MaxConcurrentRuns=1` is enforced by execution, not just represented in config.

Exit criteria:

- scheduled gap scan can generate a non-empty plan on a configured edge node
- executed plan produces durable DB changes
- advisory-driven validation uses advisory config payload, not default config only

### Stream B: Sync v2 and Advisory Lifecycle

Owner scope:

- `cmd/aima/tooldeps_integration.go`
- `internal/mcp/tools_knowledge.go`
- `internal/mcp/tools_deps.go`
- `internal/cli/knowledge.go`
- `internal/cli/scenario.go`

Primary deliverables:

1. Align client-side calls with the final advisory/scenario schema.
2. Make `knowledge.sync_pull` and `knowledge.sync_push` report protocol-level results, not just local import/export counts.
3. Filter advisory pulls by hardware and lifecycle state.
4. Distinguish protocol feedback states from UI wording.
5. Preserve idempotency for repeated pulls and feedback retries.

Exit criteria:

- edge can pull only pending advisories for its target hardware
- edge feedback updates central lifecycle deterministically
- CLI and MCP outputs reflect lifecycle state and identifiers needed for debugging

### Stream C: Central Data Model, API, and Analyzer

Owner scope:

- `internal/central/*.go`
- `cmd/central/main.go`

Primary deliverables:

1. Bring `CentralStore` schema and types in line with the design doc.
2. Add missing advisory, scenario, and analysis-run fields.
3. Implement lifecycle transitions:
   - create advisory as pending
   - mark delivered on pull
   - mark validated or rejected on feedback
   - expire stale advisories
4. Make analyzer schedule configurable and add `scenario_health`.
5. Replace duplicate `InsertAnalysisRun` status logging with real run updates.
6. Wire post-ingest delayed analysis.
7. Expose final API semantics:
   - `GET /api/v1/advisories?hardware=&status=`
   - `POST /api/v1/advisory/feedback`
   - `GET /api/v1/scenarios`
   - `GET /api/v1/analysis`

Exit criteria:

- SQLite and PostgreSQL behave consistently under the same store test suite
- analyzer runs leave coherent `analysis_runs` rows
- advisory lifecycle can be verified by integration tests

### Stream D: End-to-End Stabilization

Owner scope:

- cross-stream integration
- integration tests
- release readiness fixes

Primary deliverables:

1. Full-path tests for "new device onboarding" and "advisory validate -> feedback".
2. Failure-path tests:
   - LLM unavailable -> Tier 2 to Tier 1
   - central unavailable -> local-only exploration
   - repeated pull of same advisory
   - invalid feedback state
3. Runtime and config hardening.
4. Minimal operator docs for running central and edge in `v0.4`.

Exit criteria:

- at least one integration test covers the complete spec loop
- docs and CLI are sufficient for manual UAT

## 6. Phase Plan

### Phase 0: Contract Lock

Goal:

- settle the schema and lifecycle before parallel feature work diverges

Must finish first:

- advisory type definitions
- scenario type definitions
- `analysis_runs` update semantics
- API field names
- feedback state machine

Blocked until complete:

- final edge advisory handling
- central analyzer completion

### Phase 1: Edge Produces Real Knowledge

Goal:

- make Explorer plans execute against real inputs and leave durable outputs

Tasks:

- complete `PlanInput` gathering
- propagate hardware/config into benchmark save path
- wire harvester save/sync callbacks
- make explorer config mutable
- enforce max concurrency

### Phase 2: Sync v2 Becomes Real

Goal:

- align edge and central around the same lifecycle and payloads

Tasks:

- hardware-targeted advisory pull
- delivery state updates
- scenario fetch semantics
- feedback request normalization
- CLI and MCP result normalization

### Phase 3: Central Lifecycle and Analyzer

Goal:

- central becomes a stable producer and manager of recommendations

Tasks:

- schema migration updates
- analyzer scheduling config
- post-ingest delayed runs
- scenario health analysis
- advisory expiry and listing filters

### Phase 4: Full Loop Validation

Goal:

- prove the system behaves as one product rather than three subsystems

Tasks:

- integration suite
- local UAT checklist
- release checklist

## 7. Dependency Graph

- Stream A depends on Phase 0 contract decisions but can start on local plan execution and harvester wiring before central is done.
- Stream B depends on Phase 0 advisory contract and on Stream C API semantics.
- Stream C should lead Phase 0 because it defines the persisted contract.
- Stream D starts after A, B, and C each have their minimal vertical slice.

## 8. Testing Strategy

### 8.1 Required package tests

- `go test ./internal/agent/...`
- `go test ./internal/central/...`
- `go test ./internal/mcp/...`
- targeted `cmd/aima` tests for tooldeps and CLI flows

### 8.2 Required new test categories

- advisory lifecycle status transitions
- edge advisory validation using advisory params
- explorer schedule config mutation
- analyzer delayed post-ingest trigger
- sync pull idempotency
- scenario health output shape

### 8.3 UAT gates

The release cannot be tagged until the team can manually demonstrate:

1. edge push to central
2. central generates advisory
3. edge pulls advisory
4. explorer validates locally
5. local result changes central advisory lifecycle
6. resulting config can be promoted and discovered by future resolve paths

## 9. Risks and Countermeasures

### Risk: schema drift between edge and central

Countermeasure:

- finish Phase 0 first
- keep a single contract owner for `internal/central/store.go`

### Risk: parallel work overlaps in `cmd/aima/main.go`

Countermeasure:

- only one owner may edit `cmd/aima/main.go`
- all other streams must expose changes through non-overlapping files first

### Risk: "tests green" but product path still broken

Countermeasure:

- require at least one end-to-end integration test and one manual UAT transcript

### Risk: `v0.3` runtime configuration and `v0.4` docs diverge further

Countermeasure:

- normalize env var names before release
- document the final config matrix in release notes and operator docs

## 10. Done Definition for v0.4.0

`v0.4.0` is done when:

- P0 contract lock is implemented in code
- Streams A, B, and C are merged
- Stream D integration tests pass
- manual UAT confirms the spec loop
- release docs reflect the actual env vars and API endpoints

Until then, the codebase should be treated as `v0.4` in development, not release-complete.

## 1. Purpose

This document defines the minimum integration and UAT gates required before `v0.4.0` can be considered ready.

## 2. Merge Gates

### Gate A: Contract Stability

Must be true before edge and sync branches are merged:

- `internal/central/store.go` advisory and scenario types are stable
- central API paths and request/response fields are stable
- feedback status semantics are stable

Required checks:

- `go test ./internal/central/...`
- review of edge assumptions against final central types

### Gate B: Edge Execution Closure

Must be true before sync closure work is declared complete:

- Explorer can build a plan from non-empty runtime inputs
- Explorer-triggered benchmark execution persists knowledge artifacts
- Harvester can save notes and trigger sync callbacks
- explorer runtime config changes are live, not stubbed

Required checks:

- `go test ./internal/agent/...`
- targeted explorer tests for schedule/config/task execution

### Gate C: Sync v2 Closure

Must be true before E2E validation:

- advisory pull is hardware-aware
- lifecycle filters are respected
- feedback is idempotent enough for retry
- CLI and MCP surfaces expose advisory identifiers and status

Required checks:

- `go test ./internal/mcp/...`
- targeted `cmd/aima` tests for sync closures if available

## 3. End-to-End Scenarios

### Scenario 1: New Device Onboarding

Steps:

1. Start central with analyzer enabled.
2. Start edge with Explorer enabled and central configured.
3. Edge performs initial sync push.
4. Central stores device, sees missing coverage, and creates an advisory.
5. Edge performs sync pull and receives advisory.
6. Explorer validates advisory locally.
7. Local result is harvested and pushed.
8. Central advisory lifecycle reflects validation result.

Pass criteria:

- advisory exists on central after ingest
- advisory is visible to the target edge
- edge validation leaves at least one benchmark/config artifact
- final lifecycle is not `pending`

### Scenario 2: Tier 2 Degrade to Tier 1

Steps:

1. Start edge with cloud LLM connectivity.
2. Confirm Explorer detects Tier 2.
3. Interrupt LLM availability.
4. Trigger a new exploration event.
5. Confirm execution falls back to Tier 1 behavior.
6. Restore connectivity and confirm future work can use Tier 2 again.

Pass criteria:

- no completed work is lost
- no process restart is required
- degraded execution still produces durable artifacts where applicable

### Scenario 3: Advisory Rejection Path

Steps:

1. Central issues an advisory with a specific config.
2. Edge validates and fails or underperforms.
3. Edge pushes feedback.
4. Central updates advisory lifecycle to rejected.

Pass criteria:

- rejection reason is visible on central
- advisory is not repeatedly re-delivered as fresh pending work

## 4. Required Test Commands

Core package checks:

- `go test ./internal/agent/...`
- `go test ./internal/central/...`
- `go test ./internal/mcp/...`

Recommended broader validation before release:

- `go test ./...`
- `go vet ./...`

## 5. Manual UAT Evidence

Before release, the team should preserve:

- one terminal transcript or log snippet for each end-to-end scenario
- final advisory JSON examples
- final scenario JSON example
- final operator env var matrix for edge and central

## 6. Release Stop Conditions

Release must stop if any of these remain true:

- edge and central disagree on advisory field names
- feedback updates only `accepted bool` without lifecycle semantics
- Explorer completes runs that do not save knowledge
- analyzer writes duplicate run rows instead of updating lifecycle
- documented env vars do not match the actual binaries

## 复盘: Explorer E2E 测试 Round 4 (2026-04-07 18:55–19:10)

### 测试环境
- 设备: linux-1 (2×RTX 4090 48GB, cjwx@100.121.255.97)
- 本地引擎: sglang-kt (native), llamacpp (container), xdit-wan22 (native)
- 本地模型: 18 个 (safetensors 为主, 无 GGUF)
- 已有部署: Qwen3.5-122B-A10B on sglang-kt (PID 3920217, port 30000)
- Central: localhost:8080 (SQLite, Kimi LLM)
- LLMPlanner LLM: 本地 sglang-kt 通过 proxy (非 Kimi)

### 时间线
| 时间 | 事件 |
|------|------|
| 18:55:25 | serve 启动, explorer tier=1 |
| 18:55:50 | ProbeToolMode 探测成功, tier 升至 2 |
| 18:59:00 | LLMPlanner 生成 4 任务计划 (约 3 分钟) |
| 18:59:00–18:59:41 | Cycle 1: 4 tasks — 1 blocked (existing deploy), 1 zero-throughput (tune), 1 no-variant, 1 crashed (bad flags) |
| 19:04:50 | waitForReady 5min 超时, benchmark 报 zero throughput, not saved |
| 19:04:50–19:05:00 | Cycle 2 (RulePlanner): 3 tasks — 1 no-variant, 1 already-exists, 1 crashed (bad flags) |
| 19:10:03 | waitForReady 第二次超时, 全部完成 |

### 什么工作了
1. **Tier 升级**: 1→2 自动完成 (O4 ProbeToolMode)
2. **LLM 计划质量**: Kimi 生成的计划合理选择了本地 model+engine, 包含 sglang-kt 专属参数
3. **auto_pull=false**: 全程无引擎/模型下载
4. **Format 兼容**: llamacpp 未与 safetensors 模型配对 (B15 fix)
5. **waitForReady**: 正确等待服务就绪再 benchmark (B14 fix)
6. **Zero throughput 过滤**: 崩溃的部署 → 0 tok/s → "benchmark not saved" (B12 fix)
7. **Harvest 闭环**: 失败任务生成有意义的分析笔记 → sync 到 central
8. **RulePlanner 降级**: LLM 无响应时自动降级到规则计划 (虽然本次两个 cycle 都工作了)

### 什么没工作 (已修, 见第五轮)
1. **B16 (P2)**: auto-detect 为 sglang-kt 注入 vLLM 参数 → 进程崩溃 exit status 2 → ✅ 已修
2. **B17 (P2)**: 已有部署未检查就尝试部署 → "already starting/exists" 错误 → ✅ 已修
3. **B18 (P3)**: waitForReady 对已崩溃进程仍等 5 分钟 → ✅ 已修
4. **Catalog 缺口**: qwen3-4b, DeepSeek-R1-GPU-weight 在 sglang-kt+Ada 无 variant → ✅ B19+B20 修复 synthetic fallback

### 根因分析: 为什么全部 benchmark 为零?
核心原因是 **B16 (auto-detect 参数不匹配)** + **B17 (已有部署冲突)** + **B21 (endpoint 错误)**:
- 122B 模型: 已有 PID 占用 → deploy 被拒绝 → 无法启动新实例
- 32B/VL-8B: auto-detect 注入了 `--gpu-memory-utilization` 和 `--max-model-len`, 这是 vLLM 参数, sglang-kt 不认识 → 进程直接退出
- 其它模型: 缺少 catalog variant → resolve 失败
- 即使部署成功: benchmark 打 localhost:6188 而 sglang-kt 在 port 30000

**结论**: B16-B21 全部修复后, 第五轮 e2e 测试获得了首个真实 benchmark 数据。

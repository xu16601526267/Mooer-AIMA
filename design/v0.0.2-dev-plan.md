### 1.1 K3: L2c Golden Config Injection into Resolve Chain

**PRD Ref:** K3 (5-layer config resolution L0->L2c, P0 MVP), K5 (auto-feedback)

**Current State:**
- L0 (engine defaults) -> L2a (engine asset) -> L2b (model variant) -> L1 (user overrides) fully working
- `maybeAutoPromote()` in `cmd/aima/main.go:3117` auto-promotes benchmarks exceeding golden by >5%
- `FindGoldenBenchmark()` in `internal/sqlite.go:1199` queries golden config+benchmark
- **Gap:** Golden configs are stored in SQLite but **never read back during `Resolve()`**. The resolver only uses static YAML (L0-L2b) + user overrides (L1). L2c is a dead layer.

**Requirements:**
1. After L2b and before L1, `Resolve()` should query SQLite for the golden configuration matching `(hardware, engine, model)`.
2. Golden config values override L2a/L2b defaults but are overridden by L1 user overrides.
3. Provenance for L2c values should be `"L2c-golden"`.
4. When no golden config exists, behavior is unchanged (graceful degradation).
5. The DB query must not block offline/first-boot scenarios (DB may be nil or empty).

**Technical Design:**


- [ ] `dry_run` response shows L2c provenance when golden config is used

**Dependencies:** None (foundational, should be done first)

### 1.2 D3: Amplifier Evaluation in Engine Selection

**PRD Ref:** D3 (model format -> engine auto-selection with amplifier evaluation, P1 v1.0)

**Current State:**
- `InferEngineType()` selects engine based on format compatibility (`supported_formats`) and VRAM fit (`vram_min_mib`)
- `FormatToEngine()` dynamically maps format to engine from YAML
- **Gap:** No amplifier evaluation. Engine selection does not consider:
  - Performance gain (FlashAttention, PagedAttention, Speculative Decoding)
  - Resource boundary expansion (CPU offload via KTransformers/llama.cpp)
  - The concept of `effective_R(engine, R)` from PRD SS3

**Requirements:**
1. When multiple engines can run a model on the given hardware, rank them by **expected amplification**.
2. Amplification factors (defined in Engine YAML, not hardcoded):
   - `amplifier.performance_multiplier`: expected throughput multiplier vs baseline (e.g., vLLM=2.5x, llamacpp=1.0x)
   - `amplifier.extends_resource_boundary`: boolean, engine can use CPU/RAM for offload
   - `amplifier.effective_vram_multiplier`: when offload enabled, how much effective VRAM increases (e.g., 4.0x for KTransformers)
3. Selection priority: (a) can it run? (format + VRAM) -> (b) best amplification score
4. When VRAM is insufficient for any standard engine but an offload-capable engine exists, prefer the offload engine even if its performance multiplier is lower.

**Technical Design:**

### 1.5 S6: Effective Resource Boundary Modeling (effective_R)

**PRD Ref:** S6 (engine-expanded resource boundary modeling, P2 v1.0)

**Current State:**
- No `effective_R(engine, R)` computation
- CPU offload capability is not modeled — resolver does not consider RAM as extended VRAM

**Requirements:**
1. Define `effective_R(engine, R)` that computes the expanded resource vector when an engine supports offload.
2. Integrate into engine selection: when raw VRAM is insufficient, check if `effective_R` makes the model feasible.
3. Engine YAML declares offload capability (overlaps with 1.2 amplifier spec):
   - `amplifier.extends_resource_boundary: true`
   - `amplifier.effective_vram_multiplier: 4.0` (RAM contributes at ~25% GPU efficiency)
   - `amplifier.offload_config_key: "n_gpu_layers"` (config key that controls offload ratio)

**Technical Design:**

```
File: internal/knowledge/resolver.go

// effectiveR computes the expanded resource vector for an engine on given hardware
func effectiveR(engine *EngineAsset, hw HardwareInfo) HardwareInfo {
    expanded := hw // copy
    if !engine.Amplifier.ExtendsResourceBoundary {
        return expanded
    }
    // Add RAM contribution to effective VRAM
    ramContributionMiB := int(float64(hw.RAMTotalMiB) * (engine.Amplifier.EffectiveVRAMMultiplier - 1.0))
    expanded.GPUVRAMMiB += ramContributionMiB
    return expanded
}
```

**Integration into InferEngineType:**
```go
// Current: skip variant if vram_min_mib > hw.GPUVRAMMiB
// New: if raw VRAM insufficient, check effective_R with offload engine

    engine, _ := c.findEngine(v.Engine, hw)
    if engine != nil && engine.Amplifier.ExtendsResourceBoundary {
        effHW := effectiveR(engine, hw)

            // This engine+offload makes it feasible
            candidates = append(candidates, engineCandidate{...})

```

```go
func (p *Patrol) RunOnce(ctx context.Context) []Alert {
    var newAlerts []Alert

    // 1. Device metrics check
    result, err := p.tools.ExecuteTool(ctx, "device.metrics", nil)
    if err == nil {
        metrics := parseMetrics(result)
        if metrics.GPUTempC > p.config.GPUTempWarnC {
            newAlerts = append(newAlerts, Alert{
                Severity: "warning",
                Type:     "gpu_temp",
                Message:  fmt.Sprintf("GPU temperature %dC exceeds threshold %dC", metrics.GPUTempC, p.config.GPUTempWarnC),
            })
        }
        // ... other threshold checks ...
    }

    // 2. Deployment health check
    result, err = p.tools.ExecuteTool(ctx, "deploy.list", nil)
    if err == nil {
        deploys := parseDeploys(result)
        for _, d := range deploys {
            if d.Status == "CrashLoopBackOff" || d.Status == "Error" {
                newAlerts = append(newAlerts, Alert{
                    Severity: "critical",
                    Type:     "deploy_crash",
                    Message:  fmt.Sprintf("Deployment %s is in %s state", d.Name, d.Status),
                })
            }
        }
    }

    // 3. Store alerts
    p.mu.Lock()
    p.alerts = append(p.alerts, newAlerts...)
    p.lastRun = time.Now()
    p.mu.Unlock()

    // 4. Persist alerts to SQLite
    for _, alert := range newAlerts {
        p.db.InsertAlert(ctx, alert)
    }

    return newAlerts
}
```

### 2.3 A4: Self-Healing (Detect -> Diagnose -> Recover)

**PRD Ref:** A4 (self-healing, P1 v1.0)

**Current State:**
- K3S Pod status / Docker container status available via `deploy.list`
- Rollback mechanism exists (`agent.rollback`) but manually triggered

**Requirements:**

   - OOM (Out of Memory): container killed by OOM killer -> lower `gpu_memory_utilization` by 10% and redeploy
   - CrashLoopBackOff: 3+ restarts -> check logs for common patterns:

### 4.1 S5: ResourceSlot Abstraction

**PRD Ref:** S5 (abstract ResourceSlot with pluggable backends, P1 v1.0)

**Current State:**
- `PartitionSlot` is a flat struct with GPU/CPU/RAM fields
- No `ResourceSlot` interface; no pluggable backend abstraction
- Runtime partition passed as `PartitionRequest` directly

**Requirements:**
1. Define `ResourceSlot` interface that abstracts GPU resource allocation.
2. Pluggable backends:
   - `EngineParamSlot` — current behavior (gpu_memory_utilization) — MVP default
   - `HAMiSlot` — HAMi vGPU allocation (already partially supported in K3S runtime)
   - `MIGSlot` — NVIDIA MIG partitioning (stub, future implementation)
   - `MPSSlot` — NVIDIA MPS sharing (stub, future implementation)
3. Backend selection based on hardware capability and user preference.
4. Each backend translates the abstract slot into concrete runtime configuration.

**Technical Design:**

```
File: internal/runtime/slot.go (new)

// ResourceSlot is the abstract resource allocation interface.
type ResourceSlot interface {
    // GPUAllocation returns the GPU resource spec for container/pod configuration.
    GPUAllocation() GPUAlloc
    // CPUAllocation returns CPU limits.
    CPUAllocation() CPUAlloc
    // RAMAllocation returns memory limits.
    RAMAllocation() RAMAlloc
    // Backend returns the slot type name (e.g., "engine_param", "hami", "mig").
    Backend() string
}

type GPUAlloc struct {
    Count        int
    MemoryMiB    int
    CoresPercent int
    ResourceName string // K8s resource name
}

type CPUAlloc struct {
    Cores int
}

type RAMAlloc struct {
    MiB int
}

// EngineParamSlot implements ResourceSlot using engine config parameters (current behavior).
type EngineParamSlot struct {
    partition *knowledge.PartitionSlot
}

func (s *EngineParamSlot) GPUAllocation() GPUAlloc { ... }
func (s *EngineParamSlot) Backend() string { return "engine_param" }

// HAMiSlot implements ResourceSlot using HAMi vGPU.
type HAMiSlot struct { ... }
func (s *HAMiSlot) Backend() string { return "hami" }

// MIGSlot (stub for future)
type MIGSlot struct { ... }
func (s *MIGSlot) Backend() string { return "mig" }
```

**Integration:** Replace direct `PartitionSlot` usage in pod generation with `ResourceSlot` interface. `GeneratePod()` and Docker runtime accept `ResourceSlot` instead of raw partition values.

**Acceptance Criteria:**

- [ ] `EngineParamSlot` wraps current behavior (backward compatible)

**PRD Ref:** D4 (App dependency declaration -> auto-provision, P1 v1.0)

**Current State:**

## Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| L2c injection changes deploy behavior | High | Feature flag: `--no-l2c` to disable, gradual rollout |
| Amplifier multipliers in YAML are inaccurate | Medium | Conservative defaults (1.0x), auto-correct from benchmarks |
| Patrol generates too many alerts (noisy) | Medium | Configurable thresholds, alert dedup (same type within 1h) |
| Self-healing makes wrong diagnosis | High | Max 3 retries, then escalate to human. Audit log all actions. |
| Auto-tuning takes too long | Medium | Max config count cap, user-cancellable, concurrent benchmarks |
| Central server adds operational complexity | Low | Optional component. Edge binary works fully without it. |
| Bubble Tea dependency increases binary size | Low | Build tag: `//go:build tui` to make TUI opt-in |
